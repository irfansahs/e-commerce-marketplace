# Internationalization (i18n)

Platform-wide rules for Turkish (`tr-TR`) and English (`en-US`).

## Supported cultures (MVP)

| Code | Use |
|------|-----|
| `tr-TR` | Default |
| `en-US` | Fallback |

Clients send `Accept-Language: tr` or `Accept-Language: en` (short codes map via ASP.NET culture fallback).

## API (all microservices)

1. Call `AddMarketplaceLocalization()` + `UseMarketplaceLocalization()` from BuildingBlocks.
2. Return errors as RFC 7807 **ProblemDetails** with extension **`errorCode`** (stable, language-neutral).
3. **`title`** / **`detail`** come from `IStringLocalizer` + `.resx` per service.
4. Store user **`PreferredCulture`** when relevant (Identity register).

Example:

```json
{
  "type": "https://marketplace/errors/identity/email-taken",
  "title": "E-posta zaten kayıtlı",
  "status": 409,
  "errorCode": "IDENTITY_EMAIL_TAKEN"
}
```

## Gateway

Forward `Accept-Language` to downstream services (default Ocelot behavior for incoming headers).

## Next.js (Sprint 4+)

- [next-intl](https://next-intl.dev/) with App Router segment `[locale]` (`tr`, `en`).
- Default locale: **`tr`**. UI strings live in message JSON, not in API responses for labels.
- Auth BFF sends `Accept-Language` from active locale ([docs/WEB_AUTH.md](WEB_AUTH.md)).
- App runs in Docker (`web`); public URL `http://localhost:8080/{locale}/...`.

## Content translations (Catalog, Sprint 4+)

Product names/descriptions use **database translation rows** (or JSON), not `.resx`.

## Events

Integration events stay **locale-neutral** (ids, enums). Notification service uses `PreferredCulture` later.
